How to use your PreBlock API-Key

Everything you need to go from welcome-email to first productive template call, in your language of choice. Jump to your tier: Community · Starter · Deep integration.

1 · Base URL

All requests go to:

https://api.preblock.io

You authenticate by sending your API-key as the api_key query-parameter. There is no OAuth / bearer-token flow — a single long-lived key per license.

2 · Health check (no key needed)

curl https://api.preblock.io/api/health

Expected response:

{
  "status": "ok",
  "version": "1.0.0",
  "timestamp": "2026-02-07T14:23:11Z"
}

If the health check fails, check your firewall / outbound HTTPS on port 443. There is no IP whitelist on our side — any client with a valid key can connect.

3 · Community tier — free key, 30 calls / day

🎁 Community

You have one endpoint: /api/templates/preview. It shows how PreBlock would score a template with a given fee assumption. Read-only, no live template returned.

Call it

curl "https://api.preblock.io/api/templates/preview?api_key=YOUR_KEY&fees_sats=15000000"

Query parameters

  • api_key · required · your PreBlock key (from the welcome email)
  • fees_sats · optional · block fees to simulate in satoshis · default 15_000_000 (≈ 0.15 BTC) · min 0 · max 100_000_000_000

Response schema

{
  "license_id":      "preblock-community-caedfc41",
  "tier":            "community",
  "block_subsidy_sats":  312500000,
  "fees_sats":           15000000,
  "total_reward_sats":  327500000,
  "coinbase_split": {
    "miner_sats":       327500000,
    "vendor_sats":              0,
    "vendor_fraction":       0.00
  },
  "timestamp": "2026-02-07T14:23:11Z"
}

Community licenses have no coinbase-split — you keep 100%. This response is essentially a fee-projection calculator.

Python example

import requests
r = requests.get(
    "https://api.preblock.io/api/templates/preview",
    params={"api_key": "pb_...", "fees_sats": 20_000_000},
    timeout=10,
)
r.raise_for_status()
print(r.json())

Node.js example

const r = await fetch(
  "https://api.preblock.io/api/templates/preview?" +
  new URLSearchParams({ api_key: "pb_...", fees_sats: 20_000_000 })
);
if (!r.ok) throw new Error(`preblock: ${r.status} ${await r.text()}`);
console.log(await r.json());

Rate limit

30 requests / rolling 24 hours. Enough for hourly monitoring or a periodic fee-forecast job. Exceed it and you get HTTP 429 with a Retry-After header in seconds.

4 · Starter tier — full templates, 100 calls / minute

₿ Starter (Fix or Share)

You get both endpoints. /scored/next returns a fully optimized template ready to hand to your mining pipeline. /preview stays available for lightweight fee-projection.

The main call

curl "https://api.preblock.io/api/templates/scored/next?api_key=YOUR_KEY&adapter=gbt"

Query parameters

  • api_key · required
  • adapter · optional · output format · one of gbt (default), sv2, braiins
  • expected_fees_sats · optional · hint for pre-calculation · default 15_000_000
  • block_height_hint · optional · integer > 0 · lets our engine pre-select transaction candidates for the height you are targeting

Response schema

{
  "license_id":     "preblock-starter-fix-a7b3...",
  "tier":           "starter_fix",
  "template": {
    "version":         536870912,
    "previousblockhash": "00000000000000000000...",
    "target":          "0000000000000000000004...",
    "min_time":        1739020000,
    "curtime":         1739020123,
    "bits":            "17034dc7",
    "height":          878543,
    "coinbase_value":  325123456,
    "coinbasevalue":   325123456,
    "coinbase_split": {
      "miner_sats":    325123456,
      "vendor_sats":            0,
      "vendor_fraction":     0.00
    },
    "transactions": []
  },
  "scoring": {
    "predicted_fees_sats": 12876543,
    "confidence":          0.87,
    "generated_ms":         42
  }
}

For starter_share licenses the coinbase_split object shows the vendor cut per your contract; the split is enforced server-side.

Rate limit

100 requests / minute. Enough for one refresh per 600 ms with headroom for retries. If you poll faster we throttle you to 429 — respect the Retry-After header and back off.

Error handling — what to do

  • 200 OK — happy path, use the template
  • 401 Unauthorized — bad or missing key → double-check the value from your welcome email, no extra whitespace
  • 403 Forbidden — key is revoked or expired → email sales@preblock.io
  • 429 Too Many Requests — you hit the tier ceiling → sleep for the seconds in Retry-After, then continue
  • 5xx — server-side hiccup → exponential backoff (1s → 2s → 4s, cap at 30s), don't hammer

Production polling loop (Python)

import time, requests

def next_template(key: str, adapter: str = "gbt") -> dict:
    r = requests.get(
        "https://api.preblock.io/api/templates/scored/next",
        params={"api_key": key, "adapter": adapter},
        timeout=5,
    )
    if r.status_code == 429:
        wait = int(r.headers.get("Retry-After", "1"))
        time.sleep(wait)
        return next_template(key, adapter)
    r.raise_for_status()
    return r.json()

while True:
    tpl = next_template("pb_...")
    hand_off_to_miner(tpl["template"])
    time.sleep(0.6)   # 100/min ceiling, comfortable margin

5 · Rate-limit response format

All 429 responses follow this shape so a single client can be shared across tiers:

HTTP/1.1 429 Too Many Requests
Retry-After: 12
Content-Type: application/json

{
  "detail": "Rate limit exceeded for tier=starter_fix (100/minute)"
}

Client-side rule: never retry without honoring Retry-After. It's in whole seconds; treat missing header as 60s.

6 · Store your key safely

Your key is delivered once, in the welcome email. On our side we only store the bcrypt hash — if you lose the plaintext key, we cannot recover it and have to issue a new one (which invalidates the old).

Best practice:

  • Paste the key straight from the welcome email into a password manager (1Password, Bitwarden, KeePassXC)
  • On your server, read it from an environment variable — never commit it to git
  • If you suspect a leak, email sales@preblock.io immediately for a rotation — old key is invalidated within seconds

7 · Deeper integration paths

The direct API-key call is the fastest path. Once you're comfortable, we support three deeper patterns:

Getblocktemplate override (Bitcoin Core / Knots)

Drop-in replacement for your node's getblocktemplate RPC — no code changes on your mining side, just a config-file edit. Your miner still thinks it's talking to bitcoind; PreBlock intercepts and returns the scored template instead.

Best for: solo miners on standard bitcoind + cgminer / bfgminer / ASICSeer stacks.

DATUM adapter (OCEAN Pool)

Feed the optimized templates directly into the DATUM protocol used by OCEAN Pool. PreBlock runs as a sidecar; DATUM sees an ordinary block-template source.

Best for: OCEAN Pool participants who want the predictive boost while staying inside DATUM's censorship-resistance guarantees.

SV2 Job-Declarator (Stratum v2) — on roadmap, not shipped yet

Planned: PreBlock exposes a Job-Declaration channel over SV2 for Braiins Pool, Demand Pool, and other modern SV2 pools. Templates flow encrypted end-to-end; the Job-Declarator role stays with you.

Status: not yet implemented. Requires Rust sidecar integration with the Stratum Reference Implementation. Interested customers should mention it in their inquiry so we can prioritize.

Best for: pool operators and large solo farms on SV2 infrastructure.
